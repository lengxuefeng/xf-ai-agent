# 06 前端流程卷轴与页面终端

## 1. 前端不只是聊天窗口

这个项目的前端承担三类能力：

1. **对话输入输出**: 聊天消息交互
2. **流程卷轴可视化**: 可视化展示执行流程
3. **页面终端与运行时面板**: 受控命令执行和运行时观测

## 2. 流程卷轴的目标

流程卷轴不是简单展示日志，而是要让用户一眼看懂：

- 老板发了什么任务
- 掌柜怎么判断
- 是否有司员介入
- 是否需要审批
- 最终如何回禀

### 2.1 Workflow 事件流

前端接收后端发送的 Workflow 事件，并实时更新流程卷轴。

**事件类型**:

- `user_message_received`: 用户消息接收
- `rule_intercepted`: 规则拦截
- `workspace_prepared`: 工作区准备完成
- `memory_loaded`: 记忆加载完成
- `context_ready`: 上下文构建完成
- `tool_registry_ready`: 工具注册完成
- `agent_started`: Agent 开始执行
- `agent_completed`: Agent 执行完成
- `approval_pending`: 等待审批
- `final_report_delivered`: 最终报告交付

## 3. 运行时面板

当前卷轴已经补齐了 Runtime 相关面板：

- **Workspace**: 工作区信息和目录结构
- **Tool Call**: 工具调用记录
- **Artifact**: 运行时产物和文件
- **Exec**: 命令执行记录
- **Memory**: 记忆注入记录
- **Context**: 上下文构建信息
- **Hook**: Hook 执行记录和审批状态
- **Log**: 结构化日志流

这些面板让"模型之外发生了什么"变得可见。

### 3.1 Workflow 事件映射

前端接收 Workflow 事件并映射到流程卷轴：

| 事件 Phase | 前端展示 | 角色 |
|------------|----------|------|
| `user_message_received` | 老板下达任务 | Boss |
| `rule_intercepted` | 总管快速裁决 | Supervisor |
| `workspace_prepared` | 卷宗工位已就绪 | System |
| `memory_loaded` | 中枢调入项目与会话记忆 | System |
| `context_ready` | 上下文卷宗整理完成 | System |
| `tool_registry_ready` | 运行时工具与外部能力已装载 | System |
| `agent_started` | 司员开始执行 | Worker |
| `agent_completed` | 司员执行完成 | Worker |
| `approval_pending` | 等待老板批示 | System |
| `final_report_delivered` | 总管回禀老板 | Supervisor |

## 4. 页面终端

页面终端的目标是：

- 在页面里像 Codex 一样执行命令
- 不打开系统 Terminal
- 和聊天会话绑定
- 和工作目录绑定
- 左侧直接浏览项目目录树
- 右侧直接打开、编辑、保存文本文件

### 4.1 页面终端显示什么

- 当前工作目录
- 工作目录树
- 当前文件与编辑区
- 要执行的命令
- 命令输出
- 退出码
- 运行状态

### 4.2 页面终端实际怎么执行

页面终端不是浏览器本地执行，也不是弹出系统命令行窗口。

真实执行位置取决于后端服务运行在哪里：

- 如果后端跑在你的本机，就在你本机执行；
- 如果后端部署到服务器，就在服务器执行。

执行链路是：

1. 前端先把当前聊天会话绑定到一个工作目录；
2. 后端校验该目录是否位于允许的工作区根目录内；
3. 前端通过目录树接口和文件接口展示项目内容；
4. 用户在页面中输入命令或直接编辑文件；
5. 后端以受控子进程的方式在绑定目录或其子目录中运行命令；
6. 前端通过 SSE 流式接收命令输出。

因此，页面终端更准确地说是"页面工作台 + 后端受控执行器"。

### 4.3 页面终端不做什么

- 不允许任意目录运行
- 不默认把自然语言当成命令执行
- 不把工作目录之外的文件暴露给页面
- 不绕过后端控制

### 4.4 当前工作台模式

当前工作台已经具备这些能力：

- 绑定目录
- 浏览目录树
- 新建文件
- 新建目录
- 重命名文件或目录
- 删除文件或目录
- 打开文本文件
- 编辑并保存文本文件
- 在所选目录执行常见开发命令
- 全屏展开为更接近 Codex 的工作台视图

当前更适合的命令包括：

- `python3 app.py`
- `pytest`
- `uv run python -m unittest`
- `javac Main.java`
- `java Main`
- `npm run dev`
- `git status`

当前还不是完整的 PTY 交互式 shell。后续如果要做得更像 Codex 的原生终端，再升级成 `WebSocket + PTY` 会更合适。

### 4.5 安全约束

页面终端严格执行以下安全约束：

- **目录约束**: Path Guard 防止越界执行
- **命令白名单**: 只允许策略允许的命令
- **审批检查**: 高风险操作需要审批
- **超时控制**: 防止命令无限运行
- **输出过滤**: 防止敏感信息泄露

## 5. 前端风格

前端采用国风简约风格：

- 宣纸质感背景
- 卷轴式流程展示
- 印章式状态标记
- 墨色层次分明

避免堆砌科技风默认样式，保持产品的独特性。

## 6. 前端模式

### 6.1 老板模式

关注点：

- 整体流程可视化
- 任务执行状态
- 审批决策
- 最终结果

适合管理者和决策者使用。

### 6.2 工位模式

关注点：

- 具体执行细节
- 代码编辑
- 命令执行
- 调试信息

适合开发者和执行者使用。

前端同时支持两种模式，满足不同角色的需求。

## 7. 为什么要做成页面内终端

因为这个项目不是纯工程师工具，而是一个"流程可视化产品"。

把终端能力也放到页面里，能让老板模式和工位模式共存。

### 7.1 流程可视化

- 清晰展示"老板发令 -> 掌柜调度 -> 司员执行 -> 汇总回禀"
- 实时展示执行状态
- 支持流程回放

### 7.2 执行能力

- 受控命令执行
- 代码编辑和保存
- 文件浏览和管理
- 项目目录操作

### 7.3 产品化体验

- 国风界面设计
- 统一的交互模式
- 完整的审批流程
- 可观测的执行过程

## 8. 技术实现

### 8.1 SSE 流式响应

前端通过 Server-Sent Events (SSE) 接收后端流式推送：

- 消息流（STREAM）
- 日志流（LOG）
- Workflow 事件流（WORKFLOW）
- 审批事件（INTERRUPT）

### 8.2 状态管理

前端维护完整的状态：

- 会话状态
- 运行状态
- 审批状态
- 工作区状态

### 8.3 实时更新

通过 SSE 实时更新：

- 对话消息
- 流程卷轴
- 运行时面板
- 页面终端输出
