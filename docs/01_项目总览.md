# 01 项目总览

## 1. 这是什么项目

XF-AI-Agent 是一个"多 Agent 编排 + Harness Runtime + 前端流程可视化"的完整生产级智能体系统。

它既不是只有聊天窗口的 AI 应用，也不是只会跑代码的 Agent，而是一个兼顾以下目标的产品：

- **能对话**: 基于大模型的自然语言交互
- **能路由**: 智能判断问题类型和复杂度
- **能拆任务**: 复杂任务自动拆解为多个子任务
- **能调用工具**: 统一的工具注册和执行机制
- **能审批**: 高风险操作支持人工审批
- **能回放流程**: 完整的执行流程可追溯
- **能受控执行**: 页面终端在指定目录内安全执行

### 1.1 核心定位

这个项目可以理解为：

`上层是掌柜调度系统` + `下层是 Agent 的工作底座`

- **掌柜调度系统**: 负责判断、分发、协调、汇总
- **Agent 工作底座**: 提供文件系统、执行能力、记忆、工具等基础能力

### 1.2 产品形态

- **国风简约**: 宣纸、卷轴、印章、墨色层次
- **流程可视化**: 清晰展示"老板发令 -> 掌柜调度 -> 司员执行 -> 汇总回禀"
- **双模式支持**: 老板模式（高层决策）和工位模式（详细执行）

## 2. 解决什么问题

裸模型通常有几个硬伤：

1. **不会稳定记住项目上下文**
   - 解决方案: Memory 模块 + AGENTS.md 注入 + 结构化记忆片段

2. **不会直接操作真实工作区**
   - 解决方案: Workspace 模块 + Path Guard 安全约束

3. **不会安全地执行命令**
   - 解决方案: Exec Runtime + 命令白名单 + 审批机制

4. **不会把整个执行过程结构化暴露给用户**
   - 解决方案: Workflow 事件流 + 流程卷轴 + 运行时面板

5. **无法响应高频简单问题**
   - 解决方案: 前置规则拦截（Zero-LLM）快速响应

6. **无法支持复杂的审批流程**
   - 解决方案: Hook 机制 + Interrupt 状态 + 审批恢复

这个项目就是围绕这些问题构建的完整解决方案。

## 3. 四层架构

系统采用四层架构设计：

### 3.1 前端交互层

- **对话区**: 用户与 Agent 的自然语言交互
- **流程卷轴**: 可视化展示执行流程
- **页面终端**: 受控命令执行界面
- **运行时面板**: 实时展示运行状态和详细信息

### 3.2 接口层

- **FastAPI**: 提供 REST API 和 SSE 流式接口
- **Chat API**: 处理用户聊天请求
- **Interrupt API**: 处理审批中断和恢复
- **Terminal API**: 处理页面终端命令执行

### 3.3 编排层

- **Supervisor 编排图**: 负责任务分发和协调
- **专业 Agent**: 代码、搜索、SQL、天气、医疗、云游等专业化 Agent
- **GraphRunner**: 图执行器，使用生产者-消费者异步模型
- **前置规则拦截**: Zero-LLM 快速响应引擎
- **Agent 注册**: 统一的 Agent 注册和管理机制

### 3.4 运行时层（Harness Runtime）

- **Core**: RunContext、SessionManager、WorkflowEventBus、LiveStreamBus
- **Workspace**: 文件系统和工作区管理
- **Exec Runtime**: Bash / Python 受控执行
- **Memory**: 记忆管理和 AGENTS.md 注入
- **Tool Runtime**: Web Search / MCP / Tool Registry
- **Context Builder**: 上下文构建和注入
- **Hook Manager**: Hooks / Approval / Observability
- **Prompt Registry**: 系统提示词管理

## 4. 核心能力总览

### 4.1 多 Agent 编排

- **Supervisor 编排图**: 负责任务分发和协调
- **专业 Agent**: 代码、搜索、SQL、天气、医疗、云游等
- **GraphRunner 执行器**: 使用生产者-消费者异步模型驱动图执行
- **前置规则拦截**: Zero-LLM 快速响应，避免不必要的图执行
- **Agent 注册**: 统一的 Agent 注册和管理机制

### 4.2 Harness Runtime

- **RunContext**: 单次运行的统一上下文管理
- **SessionManager**: 运行时会话管理和状态追踪
- **WorkflowEventBus**: 工作流事件总线和持久化
- **Workspace**: 文件系统和工作区管理
- **Bash / Python 受控执行**: 限制命令白名单和工作目录
- **Memory / AGENTS.md**: 记忆管理和项目协作规则注入
- **Web Search / MCP / Tool Registry**: 统一的工具注册和执行机制
- **Context Engineering**: 历史裁剪、记忆注入、RAG 片段注入、token 预算控制
- **Hooks / Approval / Observability**: 审批 Hook、质量门禁、工具守卫、事件追踪

### 4.3 服务层

- **ChatService**: 聊天业务逻辑
- **InterruptService**: 审批中断和恢复服务
- **ChatHistoryService**: 聊天历史管理
- **SessionPool**: 预热池管理，优化 Supervisor 编译性能

### 4.4 前端展示

- **对话区**: 用户与 Agent 的交互界面
- **国风流程卷轴**: 可视化展示"老板发令 -> 掌柜调度 -> 司员执行 -> 汇总回禀"全过程
- **运行时看板**: 实时展示运行状态和会话信息
- **页面终端**: 受控命令执行界面

## 5. 技术特点

### 5.1 异步流式处理

- **生产者-消费者模型**: GraphRunner 使用生产者-消费者模型
- **后台线程执行**: 图执行在后台线程，主线程消费队列推 SSE
- **非阻塞推送**: 全程不阻塞事件循环，支持真正的并发流式推送

### 5.2 前置规则拦截

- **Zero-LLM 快速响应**: 避免不必要的图执行
- **多规则复合拦截**: 支持多个规则同时匹配
- **意图覆盖率防爆盾**: 防止复杂意图误拦截

### 5.3 审批中断恢复

- **Interrupt 挂起**: 支持 LangGraph Interrupt
- **审批恢复**: 支持状态恢复
- **状态回填**: 完整的审批工作流

### 5.4 运行时观测

- **Workflow 事件流**: 完整的工作流事件记录
- **日志流**: 结构化日志输出
- **状态快照**: 运行状态快照
- **实时监控和回放**: 支持流程回放

### 5.5 Session 管理

- **运行时会话管理**: 统一的会话生命周期管理
- **状态追踪**: 实时追踪会话状态
- **Session Pool 预热机制**: 预热 Supervisor 编译图，减少冷启动延迟
- **跨请求共享**: 跨请求共享编译好的 Supervisor 实例

### 5.6 工具注册

- **统一接口**: 统一的工具注册和调用接口
- **MCP 支持**: 支持 Model Context Protocol
- **外部工具集成**: 支持外部工具集成
- **工具执行和结果处理**: 统一的工具执行和结果处理机制

## 7. 执行流程

### 7.1 前置规则拦截流程

```text
用户请求
-> GraphRunner 接收请求
-> 前置规则拦截引擎
-> 判断是否命中规则
-> 命中规则：直接返回结果（Zero-LLM）
-> 未命中规则：进入完整图执行
```

### 7.2 完整图执行流程

```text
用户请求（未命中前置规则）
-> GraphRunner 创建 RunContext
-> SessionManager 注册会话
-> WorkspaceManager 准备工作区
-> ContextBuilder 构建上下文
-> HookManager 执行前置 Hooks
-> Supervisor 图执行开始
-> Agent 调度和执行
-> Runtime 能力调用
-> SSE 流式返回
-> HookManager 执行后置 Hooks
-> SessionManager 更新状态
-> WorkspaceManager 清理资源
```

### 7.3 审批执行流程

```text
用户明确要求执行
-> GraphRunner 接收请求
-> Supervisor 判断需要执行
-> Agent 判断为执行意图
-> Runtime Hook 触发审批
-> 进入 Interrupt 状态
-> 前端展示审批界面
-> 用户批准 / 拒绝
-> GraphRunner 按 message_id 恢复
-> Exec Runtime 在受控目录内执行
-> 输出回流前端
-> 持久化执行结果
```

## 8. 当前项目的关键价值

### 8.1 对业务型 Agent 系统友好

如果你的目标是做企业内部工作流、咨询助手、报告助手、审批助手，这套架构非常适合。

- **清晰的职责分工**: 每个 Agent 专注自己的领域
- **可扩展性强**: 容易添加新的专业 Agent
- **流程可视化**: 清晰展示执行过程
- **审批安全**: 高风险操作支持审批

### 8.2 对"老板可观测"友好

相比很多只显示最终答案的 Agent，这个项目更强调"过程可见、责任可见、风险可见"。

- **流程卷轴**: 可视化展示执行流程
- **运行时面板**: 详细展示运行状态
- **审批记录**: 完整的审批记录
- **事件追踪**: 完整的事件追踪

### 8.3 适合继续演进成平台

它现在已经不仅是单体功能，更像一个可以继续平台化的 Agent Operating Layer。

- **模块化设计**: 清晰的模块边界
- **接口抽象**: 统一的接口抽象
- **扩展机制**: Hook 扩展机制
- **工具注册**: 统一的工具注册机制

## 10. 你应该怎么理解这个项目

最容易理解的方式是：

`上层是掌柜调度系统`

`下层是 Agent 的工作底座`

二者结合起来，才是完整的生产级 Agent 系统。

- **掌柜调度系统**: 负责判断、分发、协调、汇总
- **Agent 工作底座**: 提供文件系统、执行能力、记忆、工具等基础能力

只有结合这两层，才能构建出真正可用的生产级 Agent 系统。
