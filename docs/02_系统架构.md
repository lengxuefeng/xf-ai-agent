# 02 系统架构

## 1. 总体架构

系统可以分成四层：

1. **前端交互层**
2. **API / SSE 接口层**
3. **多 Agent 编排层**
4. **Harness Runtime 层**

## 2. 主流程

```text
用户请求
-> Chat API
-> 前置规则拦截（Zero-LLM）
-> Supervisor Graph
-> Agent 调度
-> Runtime 能力调用
-> SSE 流式返回
-> 前端对话区 + 流程卷轴 + 页面终端
```

## 3. 编排层职责

编排层主要由以下部分构成：

- **Supervisor 编排图**: 负责任务分发和协调
- **专业 Agent**: 代码、搜索、SQL、天气、医疗等专业化 Agent
- **GraphRunner**: 图执行器，使用生产者-消费者异步模型驱动图执行
- **前置规则拦截**: Zero-LLM 快速响应，避免不必要的图执行
- **Agent 注册**: 统一的 Agent 注册和管理机制

这部分决定的是"谁来做"和"怎么组织任务"。

### 3.1 Supervisor 编排图

- 负责任务分发和协调
- 根据用户问题决定调用哪个专业 Agent
- 支持简单直答、单 Agent、多任务拆解等模式

### 3.2 GraphRunner 执行器

- 使用生产者-消费者异步模型
- 图执行在后台线程，主线程消费队列推 SSE
- 支持前置规则拦截
- 支持审批中断和恢复
- 支持运行时观测

## 4. Runtime 层职责

Runtime 层包含：

- **Core**: RunContext、SessionManager、WorkflowEventBus
- **Workspace**: 文件系统和工作区管理
- **Exec Runtime**: Bash / Python 受控执行
- **Memory**: 记忆管理和 AGENTS.md 注入
- **Tool Runtime**: Web Search / MCP / Tool Registry
- **Context Builder**: 上下文构建和注入
- **Hook Manager**: Hooks / Approval / Observability
- **Prompt Registry**: 系统提示词管理

这部分决定的是"Agent 怎么真正干活"。

### 4.1 Runtime 核心

- **RunContext**: 单次运行的统一上下文管理
- **SessionManager**: 运行时会话管理和状态追踪
- **WorkflowEventBus**: 工作流事件总线和持久化
- **LiveStreamBus**: 子 Agent 实时流事件总线

### 4.2 Workspace 管理

- 为每个会话、每次运行准备工作区
- 保存中间产物和最终响应
- 记录 workflow trace
- 支持 Path Guard 安全约束

### 4.3 受控执行

- 限制命令白名单
- 限制工作目录（Path Guard）
- 返回 stdout / stderr / exit code
- 支持会话管理

### 4.4 记忆管理

- 注入项目记忆（AGENTS.md）
- 注入协作规则
- 注入用户偏好
- 支持结构化记忆片段

### 4.5 工具注册

- 统一的工具注册机制
- 支持 MCP（Model Context Protocol）
- 支持外部工具集成
- 工具执行和结果处理

### 4.6 上下文工程

- 裁剪历史
- 注入记忆
- 注入 RAG 片段
- 控制 token 预算

### 4.7 Hooks 机制

- 审批 Hook
- 质量门禁
- 工具守卫
- 事件追踪
- 支持自定义 Hook

## 5. 前端职责

前端承担三类展示：

1. **聊天消息**: 用户与 Agent 的交互界面
2. **流程卷轴**: 可视化展示"老板发令 -> 掌柜调度 -> 司员执行 -> 汇总回禀"全过程
3. **页面终端与运行时面板**: 受控命令执行界面和运行状态监控

它不是纯展示层，而是部分承接了"可观测性产品化"的角色。

## 6. 技术特点

### 6.1 异步流式处理

- GraphRunner 使用生产者-消费者模型
- 图执行在后台线程，主线程消费队列推 SSE
- 全程不阻塞事件循环，支持真正的并发流式推送

### 6.2 前置规则拦截

- Zero-LLM 快速响应
- 避免不必要的图执行
- 支持多规则复合拦截
- 意图覆盖率防爆盾

### 6.3 审批中断恢复

- 支持 Interrupt 挂起
- 支持审批恢复
- 支持状态回填
- 完整的审批工作流

### 6.4 运行时观测

- Workflow 事件流
- 日志流
- 状态快照
- 实时监控和回放

### 6.5 Session 管理

- 运行时会话管理
- 状态追踪
- Session Pool 预热机制
- 性能优化

## 7. 与 Harness-first 架构的关系

你当前项目的最佳定位是：

`强编排 + 完整的 Harness Runtime`

不是推翻原来的编排系统，而是在现有编排系统下面补一层更标准的 Runtime，并且已经形成了完整的 Harness Runtime 实现体系。
