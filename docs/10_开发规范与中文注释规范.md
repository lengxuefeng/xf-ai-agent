# 10 开发规范与中文注释规范

## 1. 开发规范

### 1.1 先分层，再改代码

不要把 Runtime、Supervisor、API、前端展示逻辑混在一起改。

### 1.2 先改语义，再改 UI

例如：

- 先区分“写代码”和“执行代码”
- 再去改审批按钮和页面终端

### 1.3 高风险能力必须可观测

例如：

- Bash
- Python 执行
- 文件写入
- 外部工具调用

这些能力都必须至少有事件、状态、结果。

## 2. 中文注释规范

### 2.1 模块说明

每个核心模块顶部建议有中文说明，回答：

- 这个模块做什么
- 在整体系统里属于哪一层

### 2.2 类说明

每个核心类建议说明：

- 职责
- 主要边界

### 2.3 方法说明

每个公开方法至少说明：

- 输入
- 输出
- 副作用

### 2.4 不要写废话注释

错误示例：

- 给变量赋值
- 设置状态

正确示例：

- 为什么这里必须用 `message_id` 恢复审批
- 为什么命令目录必须校验到工作区以内

## 3. 前端注释建议

前端重点注释：

- 复杂 `useEffect`
- 流式消息处理
- 工作流面板状态同步
- 页面终端绑定逻辑

## 4. 后端注释建议

后端重点注释：

- 路由决策
- 审批恢复
- 工作目录约束
- Runtime 状态推进

## 5. 文档维护规范

1. `README.md` 只做总入口
2. 专题内容写进 `docs/`
3. 重复说明文档不保留
4. 自动生成报告不混进学习主线
