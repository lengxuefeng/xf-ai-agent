# 10 开发规范与中文注释规范

## 1. 开发规范

### 1.1 先分层，再改代码

不要把 Runtime、Supervisor、API、前端展示逻辑混在一起改。

**分层原则**:

- **Runtime 层**: 专注文件系统、执行、记忆、工具等基础能力
- **编排层**: 专注任务分发、协调、汇总
- **接口层**: 专注 API 接口和数据格式
- **服务层**: 专注业务逻辑
- **前端层**: 专注展示和交互

**修改原则**:

- 修改 Runtime 层时，不影响编排层逻辑
- 修改编排层时，不影响 Runtime 能力
- 修改接口层时，不影响业务逻辑
- 修改前端层时，不影响后端接口

### 1.2 先改语义，再改 UI

例如：

- 先区分"写代码"和"执行代码"
- 再去改审批按钮和页面终端

**语义优先**:

1. 明确业务语义
2. 设计数据结构
3. 实现业务逻辑
4. 设计 API 接口
5. 实现 UI 展示

**示例**:

```python
# 1. 先定义语义
EXECUTE_INTENT = "execute"
GENERATE_INTENT = "generate"

# 2. 再实现业务逻辑
if intent == EXECUTE_INTENT:
    # 触发审批流程
    pass
elif intent == GENERATE_INTENT:
    # 直接返回代码
    pass

# 3. 最后设计 UI
# - 根据意图显示不同的按钮
# - 审批界面根据意图动态调整
```

### 1.3 高风险能力必须可观测

例如：

- Bash
- Python 执行
- 文件写入
- 外部工具调用

这些能力都必须至少有：

- **事件**: 记录操作开始和结束
- **状态**: 记录当前状态
- **结果**: 记录操作结果
- **日志**: 记录详细日志

**示例**:

```python
# 高风险操作必须记录
def execute_command(command):
    # 1. 记录事件
    emit_workflow_event(
        phase="command_started",
        title="开始执行命令",
        summary=f"命令: {command}"
    )

    try:
        # 2. 执行命令
        result = subprocess.run(command)

        # 3. 记录结果
        emit_workflow_event(
            phase="command_completed",
            title="命令执行完成",
            summary=f"退出码: {result.returncode}"
        )

        return result
    except Exception as e:
        # 4. 记录错误
        emit_workflow_event(
            phase="command_failed",
            title="命令执行失败",
            summary=f"错误: {str(e)}"
        )
        raise
```

### 1.4 安全第一

**安全原则**:

- Path Guard 防止越界
- 命令白名单防止恶意操作
- 审批机制防止高风险执行
- 输入验证防止注入攻击

**示例**:

```python
# 必须进行路径验证
def validate_path(path, allowed_root):
    # 1. 规范化路径
    normalized = os.path.normpath(path)

    # 2. 检查是否在允许的根目录内
    if not normalized.startswith(allowed_root):
        raise SecurityError("路径越界")

    return normalized

# 必须进行命令验证
def validate_command(command, allowed_commands):
    # 1. 检查命令是否在白名单中
    if command not in allowed_commands:
        raise SecurityError("命令不在白名单中")

    return command
```

### 1.5 可观测性

**可观测性原则**:

- 结构化日志
- 指标收集
- 链路追踪
- 事件记录

**示例**:

```python
# 使用结构化日志
log.info(
    "命令执行完成",
    extra={
        "command": command,
        "exit_code": result.returncode,
        "execution_time": duration
    }
)

# 记录指标
metrics.increment("command_executed", tags={
    "command": command,
    "status": "success" if result.returncode == 0 else "failure"
})
```

## 2. 代码风格规范

### 2.1 Python 代码风格

**缩进**: 使用 2 空格缩进

```python
# 正确
if condition:
    do_something()

# 错误
if condition:
        do_something()
```

**命名**: 驼峰命名（camelCase）

```python
# 正确
getUserById()
executeCommand()

# 错误
get_user_by_id()
execute_command()
```

**函数名**: 动词开头

```python
# 正确
getUserById()
executeCommand()
validatePath()

# 错误
userId()
command()
pathValidation()
```

**注释**: 使用中文注释

```python
# 正确
# 获取用户 ID
def getUserById(user_id):
    pass

# 错误
# Get user ID
def getUserById(user_id):
    pass
```

### 2.2 文件命名规范

**Python 文件**: 小写字母 + 下划线

```python
# 正确
chat_service.py
graph_runner.py
session_manager.py

# 错误
ChatService.py
GraphRunner.py
sessionManager.py
```

**目录命名**: 小写字母 + 下划线

```
# 正确
app/runtime/core/
app/services/

# 错误
app/Runtime/Core/
app/Services/
```

## 3. 中文注释规范

### 3.1 模块说明

每个核心模块顶部建议有中文说明，回答：

- 这个模块做什么
- 在整体系统里属于哪一层

**示例**:

```python
"""
图执行器模块（GraphRunner）。

【模块职责】
- API 接口的直接消费者，负责拉起 Supervisor 图并驱动其执行。
- 将图执行过程中产生的所有事件（模型输出、工具调用、中断、错误）
  封装为标准 SSE 格式推送给前端。
- 内置前置规则拦截引擎（Zero-LLM 快速响应）。
- 负责 Interrupt 审批流的恢复和状态回填。

【设计要点】
- 生产者-消费者解耦：图执行跑在后台 daemon 线程，主线程只消费队列推 SSE，
  避免同步阻塞导致日志延迟积压。
- Supervisor 编译图按 model_config 指纹缓存，同一配置只编译一次。
- 前置规则拦截仅在输入较短时触发，避免扫描超长文本的性能损耗。
"""
```

### 3.2 类说明

每个核心类建议说明：

- 职责
- 主要边界

**示例**:

```python
class GraphRunner:
    """
    图执行器：AI 对话链路的核心调度中枢。

    职责：
    1. 管理 Supervisor 编译图缓存，同一配置只编译一次。
    2. 将同步图执行包装为可 yield 的 SSE 流，供 FastAPI StreamingResponse 消费。
    3. 拦截前置规则命中、RAG 注入、Interrupt 审批恢复等横切关注点。

    边界：
    - 不负责具体 Agent 的实现，只负责调度和事件包装。
    - 不负责 Runtime 能力的实现，只负责调用和结果处理。
    """
```

### 3.3 方法说明

每个公开方法至少说明：

- 输入
- 输出
- 副作用

**示例**:

```python
async def stream_run(
    self,
    user_input: str,
    session_id: str,
    model_config: Optional[Dict[str, Any]] = None,
    history_messages: Optional[List[Dict[str, Any]]] = None,
    session_context: Optional[Dict[str, Any]] = None,
    emit_response_start: bool = True,
) -> AsyncGenerator[str, None]:
    """
    核心流式执行器（异步生产者-消费者模型）。

    【设计说明】
    图的执行（阻塞同步）跑在后台 daemon 线程（生产者），asyncio 事件循环
    作为消费者通过 asyncio.Queue 异步取数据转为 SSE 下发给前端。
    全程不阻塞事件循环，支持真正的并发流式推送。

    Args:
        user_input:          用户当前输入文本，"[RESUME]" 表示审批恢复。
        session_id:          会话 ID，同时作为 LangGraph checkpointer 的 thread_id。
        model_config:        覆盖默认配置的模型参数字典。
        history_messages:    历史对话列表 [{user_content, model_content}, ...]。
        session_context:     结构化会话上下文（城市、用户画像等槽位）。
        emit_response_start: 是否在流开始时发送 response_start 事件。

    Yields:
        标准 SSE 格式字符串（"event: ...\ndata: ...\n\n"）。

    副作用：
        - 注册运行会话到 SessionManager
        - 创建工作区
        - 写入运行时产物
        - 持久化聊天历史
    """
```

### 3.4 不要写废话注释

**错误示例**:

```python
# 给变量赋值
user_id = 123

# 设置状态
state = "running"

# 调用函数
result = execute_command(command)
```

**正确示例**:

```python
# 使用 user_id 作为会话标识符，避免会话冲突
user_id = 123

# 将状态设置为 running，表示任务开始执行
state = "running"

# 为什么这里必须用 `message_id` 恢复审批：
# 1. message_id 是 LangGraph 的唯一消息标识
# 2. 恢复时需要精确定位到中断的消息
# 3. 否则会导致恢复到错误的节点
result = execute_command(command)
```

### 3.5 注释重点

**应该注释的内容**:

- 为什么这样做（而不是怎么做）
- 设计决策的原因
- 边界条件的处理
- 潜在的问题和风险

**不应该注释的内容**:

- 显而易见的逻辑
- 重复代码的说明
- 无意义的说明

**示例**:

```python
# 好的注释
# 使用 MD5 指纹缓存 Supervisor 图，避免相同配置重复编译
# MD5 足够用于区分不同的配置，且计算速度快
cache_key = hashlib.md5(json.dumps(config).encode()).hexdigest()

# 坏的注释
# 计算 MD5
cache_key = hashlib.md5(json.dumps(config).encode()).hexdigest()
```

## 4. 前端注释建议

前端重点注释：

- 复杂 `useEffect`
- 流式消息处理
- 工作流面板状态同步
- 页面终端绑定逻辑

### 4.1 useEffect 注释

```typescript
// 监听 SSE 流式消息
// 注意：需要在组件卸载时关闭连接，避免内存泄漏
useEffect(() => {
  const eventSource = new EventSource('/api/v1/chat/stream');

  eventSource.onmessage = (event) => {
    // 处理流式消息
    handleStreamMessage(event.data);
  };

  return () => {
    // 组件卸载时关闭连接
    eventSource.close();
  };
}, []);
```

### 4.2 流式消息处理注释

```typescript
// 处理流式消息
// 根据 event 类型分发到不同的处理器
const handleStreamMessage = (message: StreamMessage) => {
  switch (message.event) {
    case 'stream':
      // 流式内容，直接追加到聊天消息
      appendStreamContent(message.data);
      break;
    case 'workflow':
      // 工作流事件，更新流程卷轴
      updateWorkflowPanel(message.data);
      break;
    case 'log':
      // 日志事件，添加到日志面板
      appendLog(message.data);
      break;
    default:
      // 未知事件类型，记录日志
      console.warn(`Unknown event type: ${message.event}`);
  }
};
```

### 4.3 工作流面板状态同步注释

```typescript
// 更新工作流面板
// 注意：需要保持事件顺序，避免乱序导致流程展示错误
const updateWorkflowPanel = (event: WorkflowEvent) => {
  // 使用 run_id 确保事件属于当前运行
  if (event.run_id !== currentRunId) {
    return;
  }

  // 使用 phase 确定事件阶段
  const phase = event.phase;
  const existingEvent = workflowEvents.find(e => e.phase === phase);

  if (existingEvent) {
    // 更新已存在的事件
    updateEvent(phase, event);
  } else {
    // 添加新事件
    addEvent(event);
  }
};
```

### 4.4 页面终端绑定逻辑注释

```typescript
// 绑定工作目录
// 注意：需要在后端验证目录安全性，防止越界
const bindWorkspace = async (session_id: string, path: string) => {
  try {
    // 调用后端 API 绑定目录
    const response = await fetch('/api/v1/terminal/workspace/bind', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ session_id, workspace_root: path }),
    });

    const result = await response.json();

    if (result.code !== 0) {
      // 绑定失败，显示错误提示
      showError(result.message);
      return;
    }

    // 绑定成功，更新状态
    setWorkspace(result.data);
  } catch (error) {
    // 网络错误，显示错误提示
    showError('网络错误，请重试');
  }
};
```

## 5. 后端注释建议

后端重点注释：

- 路由决策
- 审批恢复
- 工作目录约束
- Runtime 状态推进

### 5.1 路由决策注释

```python
def route_to_agent(user_input: str) -> str:
    """
    根据用户输入路由到对应的 Agent。

    【路由规则】
    1. 先检查是否命中前置规则，命中则直接返回结果
    2. 未命中则判断问题类型，路由到对应的专业 Agent
    3. 如果问题复杂度低，Supervisor 直接回答

    【设计考虑】
    - 前置规则拦截可以快速响应简单问题，避免不必要的图执行
    - 专业 Agent 处理特定领域的问题，提高回答质量
    - Supervisor 处理复杂度低的问题，减少专业 Agent 调用

    Args:
        user_input: 用户输入

    Returns:
        Agent 名称或路由决策
    """
```

### 5.2 审批恢复注释

```python
async def resume_from_approval(
    session_id: str,
    message_id: str,
    decision: str,
) -> None:
    """
    从审批恢复执行。

    【恢复流程】
    1. 根据 message_id 查找审批记录
    2. 检查审批状态是否为 pending
    3. 根据 decision 更新审批状态
    4. 使用 checkpoint 恢复图执行状态
    5. 继续执行图

    【为什么必须使用 message_id】
    - message_id 是 LangGraph 的唯一消息标识
    - 恢复时需要精确定位到中断的消息
    - 否则会导致恢复到错误的节点

    Args:
        session_id: 会话 ID
        message_id: 消息 ID
        decision: 审批决策（approve/reject）
    """
```

### 5.3 工作目录约束注释

```python
def validate_workspace_path(path: str, allowed_root: str) -> str:

    验证工作目录路径是否安全。

    【为什么必须严格验证】
    - 防止用户访问系统其他目录

    Args:
        path: 用户输入的路径
        allowed_root: 允许的根目录

    Returns:
        规范化后的绝对路径

    Raises:
        SecurityError: 路径越界
    """
```

## 7. Git 提交规范

### 7.1 提交信息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型**:

- `feat`: 新功能
- `fix`: 修复 bug
- `docs`: 文档更新
- `style`: 代码格式调整（不影响功能）
- `refactor`: 重构
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建或工具相关

**Subject**:

- 简短描述（不超过 50 字符）
- 使用中文
- 使用祈使句

**示例**:

```
feat(runtime): 添加工作区管理功能

实现了工作区的创建、删除、查询功能，
支持路径验证和安全约束。

Closes #123
```

### 7.2 提交频率

**建议**:

- 功能完成一次提交
- Bug 修复一次提交
- 文档更新一次提交
- 重构一次提交

**避免**:

- 大量代码一次提交
- 混合多个功能的提交
- 无意义的提交

### 7.3 分支管理

**主分支**:

- `master`: 主分支，稳定版本

**开发分支**:

- `develop`: 开发分支
- `feature/*`: 功能分支
- `bugfix/*`: 修复分支
- `hotfix/*`: 紧急修复分支

**分支流程**:

1. 从 `develop` 创建 `feature/*` 分支
2. 在 `feature/*` 分支上开发
3. 开发完成后合并到 `develop`
4. 定期从 `develop` 合并到 `master`

- 是否有性能问题
- 是否有内存泄漏
- 是否有资源浪费

### 8.2 审查流程

1. **PR 创建**: 创建 Pull Request
2. **自动检查**: 自动运行测试和 lint
3. **代码审查**: 技术负责人审查代码
4. **修改完善**: 根据审查意见修改
5. **合并通过**: 审查通过后合并

### 8.3 审查意见

**审查意见应该**:

- 具体明确
- 有建设性
- 尊重他人
- 关注代码而不是人

**示例**:

```
好的意见：
建议将 `execute_command` 方法拆分为更小的方法，
每个方法只做一件事，提高代码可读性。

不好的意见：
这段代码写得不好，重写。
```

## 9. 总结

本规范的核心原则：

1. **分层清晰**: Runtime、编排、接口、服务、前端各司其职
2. **语义优先**: 先明确业务语义，再实现代码和 UI
3. **安全第一**: 高风险能力必须有可观测性和安全约束
4. **注释完整**: 模块、类、方法都有中文注释
5. **文档及时**: 代码修改及时更新文档

遵循这些规范，可以保证代码质量和项目可维护性。
