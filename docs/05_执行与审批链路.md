# 05 执行与审批链路

## 1. 必须先区分两种请求

### 1.1 生成代码

例如：

- 帮我写一个 Python hello world
- 给我一个 Java 冒泡排序示例

这类请求只应该返回代码，不应该直接进入执行。

### 1.2 执行代码

例如：

- 帮我执行这段 Python 代码
- 在这个目录里运行 pytest

这类请求才应该进入审批和执行。

## 2. 当前系统的完整链路

### 2.1 前置规则拦截

```text
用户请求
-> GraphRunner 接收请求
-> 前置规则拦截引擎
-> 判断是否命中规则
-> 命中规则：直接返回结果（Zero-LLM）
-> 未命中规则：进入完整图执行
```

### 2.2 完整图执行流程

```text
用户请求（未命中前置规则）
-> GraphRunner 创建 RunContext
-> SessionManager 注册会话
-> WorkspaceManager 准备工作区
-> ContextBuilder 构建上下文
-> HookManager 执行前置 Hooks
-> Supervisor 图执行开始
-> Agent 调度和执行
-> Runtime 能力调用
-> SSE 流式返回
-> HookManager 执行后置 Hooks
-> SessionManager 更新状态
-> WorkspaceManager 清理资源
```

### 2.3 审批执行链路

```text
用户明确要求执行
-> GraphRunner 接收请求
-> Supervisor 判断需要执行
-> Agent 判断为执行意图
-> Runtime Hook 触发审批
-> 进入 Interrupt 状态
-> 前端展示审批界面
-> 用户批准 / 拒绝
-> GraphRunner 按 message_id 恢复
-> Exec Runtime 在受控目录内执行
-> 输出回流前端
-> 持久化执行结果
```

## 3. 为什么审批恢复容易出错

如果恢复时没有精确找到原始中断点，就会出现：

- 已经审批，但恢复不到原任务
- 提示"未找到处于中断状态的子任务"

所以恢复必须依赖：

- `message_id`: 消息标识
- `agent_name`: Agent 名称
- `run_id`: 运行标识
- `session_id`: 会话标识
- `checkpoint`: LangGraph Checkpointer 状态

## 4. 页面终端和执行链路的关系

页面终端不是另外一套系统，而是 Runtime Exec 的前端入口。

### 4.1 Agent 执行链路

由 Agent 自己提议执行，再走审批：

```text
Agent 判断需要执行
-> 触发审批 Hook
-> 用户审批
-> 执行
```

### 4.2 页面终端执行链路

由用户直接在页面发起命令，同样走受控执行模型：

```text
用户在页面输入命令
-> Terminal API 接收请求
-> CommandSessionService 处理
-> Exec Runtime 执行
-> 流式返回输出
```

**位置**: `app/runtime/exec/`

### 4.3 共享的安全约束

两条链路共享相同的安全约束：

- Path Guard 目录限制
- 命令白名单
- 超时控制
- 输出过滤

## 5. 审批机制

### 5.1 Hook 机制

**位置**: `app/runtime/hooks/`

- **ApprovalHook**: 审批 Hook
  - 检测高风险操作
  - 触发审批流程
  - 等待用户决策

- **QualityHook**: 质量门禁
  - 检查结果质量
  - 防止低质量输出

- **ToolGuardHook**: 工具守卫
  - 检查工具调用
  - 防止恶意工具使用

### 5.2 Interrupt 机制

**位置**: `app/agent/graphs/`

- 支持 LangGraph Interrupt
- 支持状态保存和恢复
- 支持超时处理
- 支持取消操作

### 5.3 审批流程

```text
1. 检测到需要审批的操作
2. 触发 ApprovalHook
3. 进入 Interrupt 状态
4. 发送审批事件到前端
5. 前端展示审批界面
6. 用户做出决策（批准/拒绝）
7. 前端发送审批结果
8. 后端接收审批结果
9. 恢复执行或取消任务
10. 更新执行状态
```
