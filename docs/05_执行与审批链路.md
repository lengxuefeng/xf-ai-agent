# 05 执行与审批链路

## 1. 必须先区分两种请求

### 1.1 生成代码

例如：

- 帮我写一个 Python hello world
- 给我一个 Java 冒泡排序示例

这类请求只应该返回代码，不应该直接进入执行。

### 1.2 执行代码

例如：

- 帮我执行这段 Python 代码
- 在这个目录里运行 pytest

这类请求才应该进入审批和执行。

## 2. 当前系统的正确链路

```text
用户明确要求执行
-> code_agent 判断为执行意图
-> interrupt 进入审批
-> 用户批准 / 拒绝
-> GraphRunner 按 message_id 恢复
-> Exec Runtime 在受控目录内执行
-> 输出回流前端
```

## 3. 为什么审批恢复容易出错

如果恢复时没有精确找到原始中断点，就会出现：

- 已经审批，但恢复不到原任务
- 提示“未找到处于中断状态的子任务”

所以恢复必须依赖：

- `message_id`
- `agent_name`
- `subgraph_thread_id`
- `checkpoint_id`
- `checkpoint_ns`

## 4. 页面终端和执行链路的关系

页面终端不是另外一套系统，而是 Runtime Exec 的前端入口。

区别在于：

- **Agent 执行链路**  
  由 Agent 自己提议执行，再走审批。

- **页面终端执行链路**  
  由用户直接在页面发起命令，同样走受控执行模型。

## 5. 当前安全边界

1. 只能在绑定工作目录下执行
2. 只能执行策略允许的命令
3. 高风险执行仍应审批
4. 输出和产物必须可观测
