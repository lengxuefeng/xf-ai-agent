# 07 数据库与会话状态

## 1. 数据库在这个项目中的角色

数据库不只是保存用户和聊天记录，还承担：

- **会话历史**: 聊天会话和消息历史
- **审批记录**: 中断和审批状态记录
- **模型配置**: 用户自定义模型配置
- **MCP 配置**: 用户 MCP 工具配置
- **运行时状态**: 会话状态和运行状态

数据库是整个系统状态持久化的核心，支持：

- **流程回放**: 刷新页面后仍能回看流程
- **会话恢复**: 支持跨请求会话恢复
- **审批恢复**: 支持中断状态的精准恢复
- **历史追溯**: 完整的历史记录和追溯

## 2. 数据库架构

### 2.1 PostgreSQL

项目使用 PostgreSQL 作为主数据库，负责：

- 关系型数据存储
- 事务支持
- 复杂查询
- JSON 字段支持

### 2.2 Redis

项目使用 Redis 作为缓存和会话存储，负责：

- Session State 缓存
- LangGraph Checkpointer
- 热点数据缓存

## 4. 会话状态管理

### 4.1 LangGraph Checkpointer

**位置**: `app/agent/graphs/checkpointer.py`

LangGraph Checkpointer 负责保存和恢复图执行状态：

- **状态持久化**: 保存图执行状态到 Redis
- **状态恢复**: 从 Redis 恢复图执行状态
- **跨请求恢复**: 支持跨请求的状态恢复

**Checkpoint 包含**:

- 图执行状态
- 消息历史
- Agent 状态
- 工具调用状态

### 4.2 SessionManager

**位置**: `app/runtime/core/session_manager.py`

SessionManager 负责运行时会话管理：

- **注册运行会话**: 注册新的运行会话
- **更新运行状态**: 更新运行状态（pending/running/completed/failed/interrupted）
- **记录工作流事件**: 记录工作流事件
- **清理会话资源**: 清理会话资源

### 4.3 会话上下文

会话上下文包含：

- **session_id**: 会话标识
- **context_slots**: 上下文槽位（城市、用户画像等）
- **context_summary**: 上下文摘要
- **history_size**: 历史消息数量

## 5. 数据持久化策略

### 5.1 聊天消息持久化

**触发时机**: 每次对话完成后

**持久化内容**:

- 用户内容
- 模型内容
- 思考轨迹
- 工作流轨迹
- 运行时快照
- 运行时产物

**服务**: `app/services/chat_history_service.py`

### 5.2 审批记录持久化

**触发时机**: 每次进入审批状态

**持久化内容**:

- 会话 ID
- 消息 ID
- Agent 名称
- 操作名称和参数
- Checkpoint 信息
- 审批状态

**服务**: `app/services/interrupt_service.py`

### 5.3 运行时状态持久化

**触发时机**: 每次运行时

**持久化内容**:

- RunContext 信息
- 运行状态
- 工作流事件
- 工作区信息

**服务**: `app/runtime/core/session_manager.py`

## 6. 为什么 `extra_data` 很重要

当前消息表里的 `extra_data` 是前后端流程回放的关键：

- **刷新页面后仍能回看流程**: 即使刷新页面，也能重新加载历史流程
- **前端不只显示正文，还能展示运行时状态**: 前端可以根据 `extra_data` 展示完整的运行时状态

### 6.1 extra_data 结构

```json
{
  "thinking_trace": "思考轨迹文本",
  "workflow_trace": [
    {
      "phase": "user_message_received",
      "title": "老板下达任务",
      "summary": "用户输入的内容",
      "status": "completed",
      "role": "boss"
    }
  ],
  "runtime_snapshot": {
    "run_id": "运行 ID",
    "status": "completed",
    "current_phase": "final_report_delivered",
    "title": "总管回禀老板",
    "summary": "最终回答摘要"
  },
  "runtime_artifacts": [
    {
      "name": "request_context.json",
      "category": "request",
      "path": "/path/to/artifact"
    }
  ],
  "model_config": {
    "model_name": "模型名称",
    "model_config": "模型配置"
  }
}
```

## 7. 还需要重点理解的点

### 7.1 运行时快照

它描述"这一轮执行走到哪里了"。

**包含信息**:

- `run_id`: 运行标识
- `session_id`: 会话标识
- `status`: 运行状态
- `current_phase`: 当前阶段
- `title`: 当前标题
- `summary`: 当前摘要
- `error`: 错误信息
- `agent_name`: Agent 名称
- `last_workflow_event`: 最后的工作流事件
- `updated_at`: 更新时间
- `meta`: 扩展元数据

**作用**:

- 前端展示运行状态
- 支持运行状态查询
- 支持运行状态恢复

### 7.2 产物清单

它描述"这轮执行生成了哪些文件/记录"。

**包含信息**:

- `name`: 产物名称
- `category`: 产物类别（request/memory/context/artifact等）
- `path`: 产物路径
- `size`: 产物大小
- `created_at`: 创建时间

**作用**:

- 前端展示运行时产物
- 支持产物下载
- 支持产物管理

### 7.3 审批记录

它决定中断能否被精准恢复。

**包含信息**:

- `session_id`: 会话 ID
- `message_id`: 消息 ID
- `agent_name`: Agent 名称
- `status`: 审批状态
- `action_name`: 操作名称
- `action_args`: 操作参数
- `checkpoint`: Checkpoint 信息

**作用**:

- 支持审批恢复
- 支持审批历史查询
- 支持审批记录审计
